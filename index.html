  2. 删除文档

  3. 幻读的解决办法：

    1. 悲观并发控制（Pessimistic concurrency control）：在关系型数据库中被广泛的使用，假设冲突的更改经常发生，为了解决冲突我们把访问区块化。典型的例子是在读一行数 据前锁定这行，然后确保只有加锁的那个线程可以修改这行数据。
    2. 乐观并发控制（Optimistic concurrency control）：假设冲突不经常发生，也不区块化访问，然而，如果在读写过程中数据发生了变化，更新操作将失 败。这时候由程序决定在失败后如何解决冲突。实际情况中，可以重新尝试更新，刷新数据（重新读取）或者直接反馈给用 户。

分布式集群

优点：

ElasticSearch隐藏了分布式系统的复杂性，以下操作都是在底层自动完成的：

1. 将你的文档分区到不同的容器或者分片(shards)中，它们可以存在于一个或多个节点中。 
1. 将分片均匀的分配到各个节点，对索引和搜索做负载均衡。 
1. 冗余每一个分片，防止硬件故障造成的数据丢失。 
1. 将集群中任意一个节点上的请求路由到相应数据所在的节点。 
1. 无论是增加节点，还是移除节点，分片都可以做到无缝的扩展和迁移。

总结：

1. 可用于构建高可用和可扩展的系统。扩展的方式可以是购买更好的服务器(纵向扩展(vertical scale or scaling up))或者购买更多的服务器（横向扩展(horizontal scale or scaling out)）。
1. 虽然能从更强大的硬件中获得更好的性能，但是纵向扩展有它的局限性。真正的扩展应该是横向的，它通过增加节点来均摊负载和增加可靠性。

  1. 一个node（节点）就是一个ElasticSearch实例，而一个cluster（集群）由一个或多个节点组成，它们具有相同的cluster.name，协同工作，分享数据和负载。当加入新的节点或者删除节点时，集群会感知到并平衡数据。

  2. 集群中一个节点会被选举为master（主节点），它将临时管理集群级别的一些变更，例如新建或删除索引、增加或移除节点等。但主节点不参与文档级别的变更或搜索，当流量增长时，主节点不会成为集群的瓶颈

  3. 查询索引的分片情况：
    PUT /user
    {

    "settings": {
    "number_of_shards": 3,
    "number_of_replicas": 1
}

    }

  4. 问题：

    1. 集群的健康状态 yellow表示所有的主分片(primary shards)启动并且正常运行了——集群已经可以正常处理任何请求——但是复制分片(replica shards)还没有全部可用。事实上所有的三个复制分片现在都是 unassigned 状态——它们还未被分配给节点。在同一个节点上保存相同的数据副本是没有必要的，如果这个节点故障了，那所有的数据副本也会丢失。

      如何解决：
      在单一节点上运行意味着有单点故障的风险——没有数据备份。幸运的是，要防止单点故障，我们唯一需要做的就是启动另一个节点。只有一个node（节点），是无法使用副本，数据安全性得不到保障

  1. 基本名词：

    1. number_of_shards：主分片
    2. number_of_replicas：复制分片

初识ElasticSearch

  1. 基本概念：索引（indexing）、搜索（search）、聚合（aggregations）

在ElasticSearch中，文档归属于一种类型（Type）,而这些类型存在索引（index）中，对传统的关系型数据库对比

Relational Db -> Databases ->Tables -> Rows ->Columns

ElasticSearch -> Indices(索引) -> Types(类型) -> Documents(文档) -> Fields

ElasticSearch集群中可以包含多个Indices(索引，数据库)，每一个Indices（索引）可以包含多个Types(类型，表)，每一个Types（类型）包含多个文档（Documents，行），然后每个文档包含多个Fields（字段，行）

  1. 简单语法：

    1. 插入：

      PUT /megacorp/employee/1
{
    "first_name":"Zhang",
    "last_name":"San",
    "age":25
}

      其中：megacorp为Index（索引名），employee为Type（类型），文档id（员工id）

    2. 简单查询：

      1. GET /megacorp/employee/1

        根据index（索引名）、Type（类型）以及员工id查询到张三的信息，响应的内容中包含一些文档的元信息，如下所示

        {
    "_index": "megacorp",
    "_type": "employee",
    "_id": "1",
    "_version": 1,
    "found": true,
    "_source": {
            "first_name":"Zhang",
            "last_name":"San",
            "age":25
        }
}

      2. GET /megacorp/employee/_search

        查询指定Index以及Type查询所有员工的数据，使用_search代理原来的文档id，默认会搜索会返回前10个结果。

        {

      3. GET /megacorp/employee/_search?q=last_name:San

        在请求中依旧使用 _search 关键字，然后将查询语句传递给参数 q=，可以得到姓氏为San的结果

  2. DSL查询（Query DSL 复杂查询）

    1. 得到姓氏为San的记录

    1. 搜索所有名字有a的员工：

    1. 我们想要查询同时包含”rock”和”climbing”（并且是相邻的）的员工记录,要将 match 查询变更为 match_phrase 查询

基本名词：

Index：

1. 索引（名词）：存储相关文档的地方，复数是indices或indexes。
2. 索引（动词）：索引一个文档，表示把一个文档存储到索引（名词）里，以便它可以被检索或者查询。与sql中的insert关键字很像，差别是当文档存在，新的文档将覆盖旧的文档。
3. 倒排索引：传统数据库是为特定的列增加一个索引，但ElasticSearch和Lucene使用一种叫做倒排索引（inverted index）的数据库结构来达到相同目的（默认情况下，文档中所有字段都会被索引，拥有一个倒排索引）

依赖注入

ASP.NET Core 支持依赖关系注入 (DI) 软件设计模式，这是一种在类及其依赖关系之间实现控制反转 (IoC) 的技术

依赖注入是控制反转思想的一种实现模式。一种消除类之间依赖关系的设计模式。重要的方式就是将依赖对象的创建和绑定转移到被依赖对象类的外部来实现。

优点：

  1. 轻松管理类及类之间的依赖；
  2. 减少代码耦合性；
  3. 提高代码维护性和可扩展性；
  4. 由容器统一创建对象和管理生命周期

依赖注入的三种方式：属性注入、方法（接口）注入、构造函数注入

  1. 服务的生命周期：Singleton、Scoped、Transient

  2. NetCore中依赖注入关键的类型：

    1. ServiceDescriptor：服务注册的信息，如服务类型、实现类型、实例类型、生命周期、实例化对象工厂等

    2. IServiceCollection：负责存储注册的服务，可以通过其扩展方法进行服务注册，是ServiceDescriptor的集合

    3. IServiceProvider：容器，是IServiceCollection创建出来的，用来提供实例的

    4. IServiceScope：表示一个容器的子容器的生命周期

      ServiceDescriptor类的属性

  3. 扩展提供的使用方式：

    1. services.AddSingleton<IHttpContextAccessor, HttpContextAccessor>();
    2. services.AddSingleton(new HttpContextAccessor());
    3. services.AddSingleton(typeof(IHttpContextAccessor), provider =>
      {
      return new HttpContextAccessor();
      });
    4. services.AddSingleton(typeof(IHttpContextAccessor),typeof(HttpContextAccessor));
    5. services.TryAddSingleton<IHttpContextAccessor, HttpContextAccessor>();//已存在该类型的服务，则不添加，该方法适用于同一个服务已存在的服务不在添加的场景
    6. services.TryAddEnumerable(new ServiceDescriptor(typeof(IHttpContextAccessor),typeof(HttpContextAccessor),ServiceLifetime.Singleton));//如果已存在对应的服务则不添加，适用于为同一个服务添加多个不同的实现的场景
    7. services.TryAddTransient(typeof(IHttpContextAccessor),typeof(HttpContextAccessor));//已存在该类型的服务，则不添加，该方法适用于同一个服务已存在的服务不在添加的场景

重要概念：

依赖倒置原则（DIP）：一种软件架构设计的原则（抽象概念）。

控制反转（IoC）：一种反转流、依赖和接口的方式（DIP的具体实现方式）。

依赖注入（DI）：IoC的一种实现方式，用来反转依赖（IoC的具体实现方式）。

IoC容器：依赖注入的框架，用来映射依赖，管理对象创建和生存周期（DI框架）。

扩展篇

  1. 结果缓存分为四种：Root、Scope、Dispose、None

  2. 每个提供六种可得到其实现：Factory、IEnumerable、Constructor、Constant、ServiceProvider、ServiceScopeFactory

  3. CallSiteFactory、CallSiteRuntimeResolver、ExpressionResolverBuilder是目标服务实例化的核心实现：

    CallSiteFactory：解析和缓存目标服务的实例化上下文
CallSiteRuntimeResolver：使用反射完成目标服务的实例化
ExpressionResolverBuilder：使用表达式树得到目标服务的实例化的前置委托

  4. ServiceProvider通过特殊的委托完成了目标服务实例化方式的替换：

  5. 初次调用GetService()时，首先通过DynamicServiceProviderEngine返回了委托，该委托被存储到字典 RealizedServices中；接着该委托被第1次执行，通过CallSiteRuntimeResolver完成目标服务的实例化；

  6. 再将调用GetService()时，直接得到缓存的委托并同样完成目标服务的实例，同时通过一个额外的 Task，通过ExpressionResolverBuilder 使用表达式树重新生成委托，并操作字典RealizedServices，替换初次调用生成委托；

  7. 后续调用GetService()时，字典RealizedServices查找到的是已经替换过的使用表达式树生成的委托。

控制反转

IoC是什么？

IoC控制反转（Inversion of control）：是一种是面向对象编程中的一种设计原则，用来减低计算机代码之间的耦合度。其基本思想是：借助于“第三方”实现具有依赖关系的对象之间的解耦，是一种设计思想。

引进了中间位置的“第三方”，也就是IOC容器，使得A、B、C、D这4个对象没有了耦合关系，齿轮之间的传动全部依靠“第三方”了，全部对象的控制权全部上缴给“第三方”IOC容器。对象A依赖对象B，但通过引入IOC容器后，主动创建对象变成了被动创建，由主动行为变为了被动行为，控制权颠倒过来了，这就是“控制反转”这个名称的由来。

为什么要使用IoC？

  1. 解耦合（解决了调用者和被调用者依赖耦合关系）
  2. 为了使用较少的代码来实现相应的功能

IOC容器的职责：

依赖查找是IoC的职责,决定加载哪个提供者
依赖注入是把控制权交给提供者

调用者根据需要在运行时刻创建提供者,而创建方法由IoC提供
调用者不负责如何创建,创建什么,也不需要知道创建了什么,
调用者要做的仅仅是传递一个接口给提供者,
所以说是:
调用者实现了接口,提供者使用接口
(比如说,你开放一个事件或者委托,提供者就能在你需要的时刻自动启动,甚至完全接管)

  1. 名词：

    依赖倒置原则（DIP）：一种软件架构设计的原则（抽象概念）。

    控制反转（IoC）：一种反转流、依赖和接口的方式（DIP的具体实现方式）。

    依赖注入（DI）：IoC的一种实现方式，用来反转依赖（IoC的具体实现方式）。

    IoC容器：依赖注入的框架，用来映射依赖，管理对象创建和生存周期（DI框架）

DI与Factory模式：

是软件结构控制反转（IoC）的两种实现。
    依赖注入的名称暗示该引用已注入或提供给您的代码。使用Factory模式，您的代码必须请求引用，以便您的代码访存该对象。两种实现都删除或分离了代码与代码所使用的对象引用的基础类或类型之间的链接。
    值得注意的是，可以编写Factory模式（或者实际上是抽象Factory模式，即返回新工厂并返回对象引用的工厂的抽象工厂模式），以便在运行时动态选择或链接到所请求对象的类型或类。这使得它们与IoC的另一种实现的Service Locator模式非常相似（甚至比DI更是如此）。

1. MakeGenericType函数，来为泛型方法和泛型类指定（泛型的）类型

using System;
using System.Reflection;

namespace RFTest
{
    //类ReflectionTest中定义了一个泛型函数DisplayType和泛型类MyGenericClass
    class ReflectionTest
    {
        //泛型类MyGenericClass有个静态函数DisplayNestedType
        public class MyGenericClass<T>
        {
            public static void DisplayNestedType()
            {
                Console.WriteLine(typeof(T).ToString());
            }
        }

        public void DisplayType<T>()
        {
            Console.WriteLine(typeof(T).ToString());
        }
    }

    class Program
    {
        static void Main(string[] args)
        {
            ReflectionTest rt = new ReflectionTest();

            MethodInfo mi = rt.GetType().GetMethod("DisplayType");//先获取到DisplayType<T>的MethodInfo反射对象
            mi.MakeGenericMethod(new Type[] { typeof(string) }).Invoke(rt, null);//然后使用MethodInfo反射对象调用ReflectionTest类的DisplayType<T>方法，这时要使用MethodInfo的MakeGenericMethod函数指定函数DisplayType<T>的泛型类型T

            Type myGenericClassType = rt.GetType().GetNestedType("MyGenericClass`1");//这里获取MyGenericClass<T>的Type对象，注意GetNestedType方法的参数要用MyGenericClass`1这种格式才能获得MyGenericClass<T>的Type对象
            myGenericClassType.MakeGenericType(new Type[] { typeof(float) }).GetMethod("DisplayNestedType", BindingFlags.Static | BindingFlags.Public).Invoke(null, null);
            //然后用Type对象的MakeGenericType函数为泛型类MyGenericClass<T>指定泛型T的类型，比如上面我们就用MakeGenericType函数将MyGenericClass<T>指定为了MyGenericClass<float>，然后继续用反射调用MyGenericClass<T>的DisplayNestedType静态方法

            Console.ReadLine();
        }
    }
}

六大原则

  1. ISP（接口隔离原则）：一个类对另外一个类的依赖应建立在最小的接口上（客户端不应该被迫依赖于它不适用的方法）

    Tip：类不应该依赖于它不需要的接口，一个类对另外一个类的依赖应该建立在最小的接口上。

  2. SRP（单一职责原则）：单一职责原则规定一个类有且仅有一个引起它变化的原因

    1. 代码重用性：相同功能的代码，不用多次编写

    2. 可读性：编程规范性，便于其他程序员的阅读和理解

    3. 可扩展性：当需要增加新的功能时，非常的方便—可维护

    4. 可靠性：当我们新增功能以后，对原来的功能没有影响

    5. 使程序呈现高内聚、低耦合的特性

      Tip：单一职责使得类的复杂度降低、提高了复用性以及可维护性

  3. DIP（依赖倒置原则）：要面向接口编程，不要面向实现编程

    1. 高层模块不应该依赖底层模块，二者应该依赖其抽象（依赖接口或抽象类，不要依赖具体的子类）

    2. 抽象不应该依赖细节，细节应该依赖抽象

    3. 依赖倒转(倒置)的中心思想是面向接口编程

    4. 使用接口或抽象类的目的是制定好规范，而不涉及任何具体的操作，把展现细节的任务交给他们的实现类去完成

    5. 依赖倒转原则是基于这样的设计理念：相对于细节的多变性，抽象的东西要稳定的多。以抽象为基础搭建的架构比以细节为基础的架构要稳定得多

      Tip：面向接口编程，不要面向实现编程，底层模块尽量为抽象类或者接口

  4. LoD（迪米特法则，最少只是原则）：只与你的直接朋友交谈，不跟“陌生人”说话

    1. 一个对象应该对其他对象保持最少的了解

    2. 类与类关系越密切，耦合度越大

    3. 迪米特法则(Demeter Priciple)又称最少知道原则，即一个类对自己依赖的类知道的越少越好。也就是说，对于被依赖的类不管多么复杂，都尽量将逻辑封装在类的内部。对外除了提供的public方法，不对外泄露任何消息

    4. 只与直接朋友通信

    5. 直接的朋友：每个对象都会与其他对象有耦合关系，只要两个对象之间有耦合关系，我们就说这两个对象之间是朋友关系。耦合的方式越多，依赖、关联、组合、聚合等，其中，我们称出现成员变量，方法参数，方法返回值中的类为直接朋友，而出现在局部变量中的类不是直接的朋友。也就是说，陌生的类最好不要以局部变量的形式出现在类的内部。

      Tip：尽量减少与其他类的相互依赖

  5. LSP（里氏替换原则）：子类可以扩展父类的功能，但不能改变父类原有的功能

    1. 提高代码复用性、可扩展性

    2. 侵入性（只要继承就必须拥有父类的所有方法和属性，一定程序约束子类）；耦合性增强（父类修改时需要考虑所有子类，很可能造成子类功能出现故障）

      Tip：遵循里氏替换原则，在子类中尽量不要重写父类的方法

  6. OCP（开闭原则）：软件实体应当对扩展开放，对修改关闭（编程中最基础、最重要的设计原则）

    1. 一个软件实体如类，模块和函数应该对扩展开放（对于提供方来说），对修改关闭（对于使用方来说）。用抽象构建框架，用实现扩展细节。

    2. 当软件需要变化时，尽量通过扩展软件实体的行为来实现变化，而不是通过修改已有的代码来实现变化。

    3. 编程中遵循其它原则，以及使用设计模式的目的就是遵循开闭原则。

      Tip：对修改关闭，对扩展开放

使用EFCore针对枚举类型筛选时的坑

使用EFCore针对枚举类型筛选时的坑

EF条件中默认不支持枚举类型，枚举类型无法转换为sql语句去查询，最后的结果是在内存中查询的，导致最后得到的数据以及性能都有差距。例如：

/// <summary>
/// 文件类型
/// </summary>
public enum FileTypeEnum
{
    /// <summary>
    /// 未知
    /// </summary>
    [Description("未知")] Unknow = 0,

    /// <summary>
    /// 图片
    /// </summary>
    [Description("图片")] Img = 1,

    /// <summary>
    /// 音频
    /// </summary>
    [Description("音频")] Audio = 2
}

/// <summary>
/// 文件类
/// </summary>
public class Files : AggregateRoot<long>
{
    /// <summary>
    /// 文件名称
    /// </summary>
    public string Name { get; set; }

    /// <summary>
    /// 文件真实地址
    /// </summary>
    public string Path { get; set; }

    /// <summary>
    /// 文件类型
    /// </summary>
    public FileTypeEnum FileType { get; set; }

    /// <summary>
    /// 创建时间
    /// </summary>
    public DateTime CreateTime { get; set; }
}


/// <summary>
/// 映射文件配置
/// </summary>
public class FileMap : IEntityMap<Files>
{
    public void Map(EntityTypeBuilder<Files> builder)
    {
        builder.ToTable("files");
        builder.HasKey(t => t.Id);
        builder.Property(t => t.Id).HasMaxLength(50);

        builder.Property(t => t.Name).HasMaxLength(50);
        builder.Property(t => t.Path).HasMaxLength(1000);
        builder.Property(t => t.FileType);
        builder.Property(t => t.CreateTime);
    }
}

下面是查询的实现类：

public class FileRepository : RepositoryBase<Files, long>, IPerRequest, IFileRepository
{
    public FileRepository(IUnitOfWork unitOfWork) : base(unitOfWork)
    {

    }

    /// <summary>
    /// 得到指定文件类型的文件列表
    /// </summary>
    /// <returns></returns>
    public List<Files> GetList(FileTypeEnum fileType,int pageIndex,int pageSize)
    {
        return Dbcontext.Set<Files>().Where(x => x.FileType == fileType).Skip((pageIndex-1)*pageSize).Take(pageSize).ToList();
    }
}

这样的方法是我们经常用到的，平时也不注意，但是这样的代码有很大的性能问题，我们的本意是查询指定文件类型的文件列表，但是最后生成的sql语句却是下面的：

SELECT `x`.`Id`,  `x`.`CreateTime`,`x`.`FileType`, `x`.`Name`, `x`.`Path` FROM `deyouyun_files` AS `x` Limit 0,20

虽然最后得到的数据仍然是指定文件类型的文件列表，但是为什么生成的sql语句却没有条件呢，这是因为数据库不支持枚举类型，所以，但很明显我们不希望如此，那么我们可以按照以下方式去调整Map映射即可

/// <summary>
/// 映射文件配置
/// </summary>
public class FileMap : IEntityMap<Files>
{
    public void Map(EntityTypeBuilder<Files> builder)
    {
        builder.ToTable("files");
        builder.HasKey(t => t.Id);
        builder.Property(t => t.Id).HasMaxLength(50);

        builder.Property(t => t.Name).HasMaxLength(50);
        builder.Property(t => t.Path).HasMaxLength(1000);
        builder.Property(t => t.FileType).HasConversion(new EnumToNumberConverter<FileTypeEnum,int>());
        builder.Property(t => t.CreateTime);
    }
}

调整完成后再次尝试运行即可发现，生成的sql语句如下所示：

SELECT `x`.`Id`,  `x`.`CreateTime`,`x`.`FileType`, `x`.`Name`, `x`.`Path` FROM `deyouyun_files` AS `x` WHERE `x`.`FileType` = 1 Limit 0,20
